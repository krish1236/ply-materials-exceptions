Design notes

The exception layer

A small backend that sits on top of a contractor inventory feed and answers three questions for every scheduled job: is it at risk, why, and what to do. The sections below cover the design choices that matter.

The framing that drove every other choice

An inventory system is a belief about where parts are. Every number in that belief was last touched by a human, and people miscount, forget to scan, and don't always close the loop on transfers. So the system has to reason about trust in its own state, not only about counts.

That is why the central abstraction is not "stock." It is confidence.

Event log as the only thing that's true

Every count, every PO state, every price baseline is a projection over an append-only log. Nothing writes to projection tables except the projector, replaying the log.

The idempotency guarantee lives at the storage layer, not the app: UNIQUE (source, external_id) on events, combined with INSERT ... ON CONFLICT DO NOTHING RETURNING id. If rows affected is 1, new; if 0, the duplicate has already been accepted and we return 200. No locks, no race between a check and an insert.

src/main/java/com/ply/exceptions/events/EventStore.java

Webhooks are hostile input

The ingestion path verifies the signature before parsing any JSON. The compare uses MessageDigest.isEqual so timing is constant regardless of where the bytes diverge. A timestamp window bounds replay. 401 on any failure, with no body. Malformed payloads land in a dead_letters row with the reason and the raw body so an operator can fix the upstream and replay.

Confidence as a pure function

The scorer takes recency of last scan, activity since, trust of the last event's source, and a conflict term. Weights sum to one, output is always in [0, 1]. No training, no model. Every weight is a constant in the source file.

src/main/java/com/ply/exceptions/confidence/ConfidenceScorer.java

The activity term halves every two intervening events (Math.pow(0.5, n / 2.0)), which matches how warehouse discipline tends to decay. Recency decays linearly over 14 days. Source trust is a small table: scan and receive at 0.9, transfer at 0.7, adjustment at 0.5, inferred at 0.3.

Rules as pure functions over a materialized view

Each rule implements one method: List<Candidate> evaluate(JobRiskView view). No I/O, no DB access inside the rule. The builder materializes the view once per job; rules are pattern matches that attach a probable cause, a suggested action, and references to the events that triggered them.

src/main/java/com/ply/exceptions/rules/JobRiskView.java
src/main/java/com/ply/exceptions/rules/RulesEngine.java

Six rules ship: missing part risk, count likely stale, unrecorded usage, untouched too long, PO will miss job date, reorder price spike. Two are deferred and documented in the README. Transfer not completed and conflicting location state both require correlation across events and do not fit cleanly into a view built for each job. Forcing them in would have bent the architecture for no good reason.

Alert identity is deterministic

Each candidate carries a dedupe_key like COUNT_LIKELY_STALE:job_4721:BRK-20A:truck_8. The alerts table has a unique index on that key. Upserts collapse repeated evaluations into the same row and bump last_seen_at. On a pass where a candidate no longer fires, a batch UPDATE alerts SET status = 'auto_resolved' WHERE last_seen_at < run_start AND status IN ('open', 'acknowledged') closes the stragglers.

src/main/java/com/ply/exceptions/alerts/AlertService.java

One intentional stickiness: if a human marked an alert resolved, the next firing of the same key does not reopen it. Reopens only happen for alerts that were auto_resolved. A person's decision outranks a rule's.

Resilience patterns kept minimal

A circuit breaker per source, with opened_at persisted in Postgres. Opens after five consecutive DataAccessException errors from the event store append and cools down for 60 seconds. The breaker fronts the webhook controller so a genuinely broken dependency does not cascade into bad events landing in the log.

src/main/java/com/ply/exceptions/resilience/CircuitBreaker.java
src/main/java/com/ply/exceptions/resilience/ReplayService.java

Replay is the primary recovery path. It truncates projections, resets the projector offset, reapplies every event in order, and reruns rules. On the current dataset it takes under a second. The event log is never touched.

Trade-offs accepted

  • Postgres is the event log. At real volume this becomes Kafka or Kinesis. At current volume an extra broker is noise.
  • Single tenant. Every table would get a tenant_id column and a filter at the edge. The product shape does not change.
  • Thymeleaf, not React. Three screens rendered from the server are enough; a JS framework would add a build pipeline and client state without improving what the user sees.
  • Scheduled rules every 30 seconds rather than fired on each new event. The latency is fine for the morning flow of an operations manager; firing on each event is a next step if the latency ever matters.
  • Synthetic webhooks. A production integration needs sandbox credentials. The envelope shape matches the pattern real sources use, so an adapter is one file per source.

What the system does not do

It is not an inventory platform. It is not a purchasing tool. It does not forecast, does not shop for prices, does not integrate with field service systems. It does one thing: for every job scheduled in the next two weeks, it decides whether the job is at risk, why, and what to do.
